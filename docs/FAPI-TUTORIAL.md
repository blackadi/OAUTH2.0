## Quick Start (5-Minute FAPI 2.0 SP)

Get a working FAPI 2.0 Security Profile flow end-to-end in 5 minutes.

### 1. Enable FAPI 2.0 Security Profile

In [Authlete Console](https://console.authlete.com/):

| Setting | Path | Value |
|---------|------|-------|
| FAPI Profile | Service Settings → Endpoints → Advanced → FAPI | Check `FAPI2_SECURITY` |
| DPoP Nonce | Service Settings → Tokens and Claims → Advanced → DPoP Token | Require Nonce: `true`, Duration: `3600` |

### 2. Create a scope with fapi2=sp attribute

Authlete triggers FAPI 2.0 SP validation **per-request** based on scope attributes. The `fapiModes` setting enables the infrastructure, but individual requests only get FAPI enforcement when they include a scope with the `fapi2=sp` attribute.

1. **Service Settings → Tokens and Claims → Advanced → Supported Scopes**
2. Click **Create** and enter a scope name (e.g. `fapi_scope`)
3. Under **Scope Attributes**, add an entry:
   - **Key**: `fapi2`
   - **Value**: `sp`
4. Click **Save**

### 3. Create a confidential client with PRIVATE_KEY_JWT

FAPI 2.0 SP requires `private_key_jwt` (or `tls_client_auth`) for client authentication.

1. **Clients → Create** → Client Type: `Confidential`
2. Token Auth Method: `PRIVATE_KEY_JWT`
3. Grant Types: `AUTHORIZATION_CODE`, `REFRESH_TOKEN`
4. Redirect URIs: `http://localhost:3001/callback`
5. Under **JWK Set**, paste a placeholder key (you will replace it with the key generated by the SPA wizard):
   ```json
   {"keys":[{"kty":"EC","crv":"P-256","kid":"placeholder","x":"","y":"","alg":"ES256","use":"sig"}]}
   ```
6. Enable **PAR (Pushed Authorization Requests)** requirement
7. Click **Save** and note the `clientId`

After generating keys in the SPA wizard (Step 4 below), update the **JWK Set** with the real public key.

### 4. Start the servers

```bash
docker compose up -d redis          # optional, for session storage
npm --prefix server run dev          # Express on :3000
npm --prefix client run dev          # SPA on :3001
```

### 5. Run the wizard

1. Open `http://localhost:3001` → click **FAPI 2.0 Security Profile** in sidebar
2. In the **FAPI 2.0 SP Test Flow** card:
3. **Setup**: Enter your `clientId`, redirect URI, and scope including `fapi_scope`
4. Click **Generate Client Auth Key** → copy the displayed JWK Set
5. In Authlete Console, update the client's **JWK Set** with this key
6. Click **Generate DPoP Key**
7. **Step 1**: Click **Push PAR** → creates a pushed authorization request with `private_key_jwt` + DPoP
8. **Step 2**: Click **Open Authorize Page** → log in as `admin` / `password` → consent
9. You'll be redirected to the callback page — tokens are stored automatically
10. Navigate back to the FAPI section
11. **Step 3**: Click **Call Userinfo with DPoP** → fetches userinfo with DPoP-bound token

### 6. Verify it's working

```bash
curl http://localhost:3000/api/fapi/config
# Expected: {"mode":"sp","dpopEnabled":true,"requiredClientAuth":"PRIVATE_KEY_JWT",...}
```

Your `access_token` has `token_type: "DPoP"` and is cryptographically bound to your browser key. See [Part 6](#part-6-failure-demonstrations) for token theft demos.

## Part 1: What is FAPI 2.0?

**FAPI** (Financial-grade API) is a security profile built on top of OAuth 2.0 and OpenID Connect. It was originally designed for the financial industry but is now widely adopted for any high-security API.

### FAPI 2.0 Security Profile

The **Security Profile** (SP) is FAPI 2.0's primary mode. It requires:

| Requirement | Why | RFC/Spec |
|-------------|-----|----------|
| **PAR** (Pushed Authorization Requests) | Prevents authorization request tampering by requiring the client to push parameters directly to the AS | RFC 9126 |
| **PKCE** (S256) | Prevents authorization code interception | RFC 7636 |
| **Sender-constrained tokens** (DPoP or mTLS) | Binds the access token to the legitimate holder — a stolen token is useless | RFC 9449 (DPoP) |
| **private_key_jwt** client auth | Strong asymmetric client authentication using signed JWTs | RFC 7523 |
| **No refresh token rotation** | FAPI 2.0 prohibits rotation (unlike FAPI 1.0) | FAPI 2.0 §5.3.2.1 |
| **scope required** | Rejects requests without a scope parameter | RFC 6749 §3.3 |
| **iss response parameter** | Prevents mix-up attacks across multiple AS | RFC 9207 |

### FAPI 2.0 Message Signing

Adds signed request objects (JAR), JARM (JWT Secured Authorization Response Mode), and introspection response signing. This tutorial covers the Security Profile only.

### How Authlete enforces FAPI

Authlete uses a **two-layer** model:

1. **Service-level**: The `fapiModes` array enables FAPI infrastructure (what features Authlete is prepared to enforce)
2. **Request-level**: A scope attribute (`fapi2=sp`) triggers actual per-request enforcement

Without the `fapi2=sp` scope attribute in the request, Authlete treats the flow as standard OAuth even with `FAPI2_SECURITY` enabled. The scope attribute is what tells Authlete "this specific request must comply with FAPI rules."

### DPoP: how it fits into FAPI

DPoP (Demonstration of Proof-of-Possession, RFC 9449) is the mechanism FAPI 2.0 SP uses for **sender-constrained tokens**. It is not a separate feature — it is FAPI's answer to the question "how do we prevent token theft?"

Within FAPI 2.0 SP:
- **PAR** handles request integrity (replacing signed request objects)
- **PKCE** handles code interception prevention
- **private_key_jwt** handles client authentication
- **DPoP** handles token binding (ensuring only the client that obtained the token can use it)

Authlete processes DPoP transparently as part of its token and API validation pipeline. The server forwards DPoP headers (`DPoP`, `htm`, `htu`) from the HTTP request to Authlete, which validates them. No additional server logic is needed — DPoP validation is a built-in Authlete capability that FAPI enforcement activates.

### How the server determines FAPI mode

The server calls `authleteApi.service.get()` and inspects the `fapiModes` array:

| `fapiModes` | Mode Reported | PAR Required | Notes |
|---|---|---|---|
| `[]` (empty) | `disabled` | No | Standard OAuth/OIDC |
| `["FAPI2_SECURITY"]` | `sp` | Yes | Uses PAR instead of signed request objects. PKCE + DPoP required. |
| `["FAPI2_SECURITY", "FAPI2_MESSAGE_SIGNING"]` | `ms` | Yes | Adds JARM + introspection signing + signed request objects with `nbf` |

## Part 2: Authlete Console Setup

All FAPI configuration happens in the [Authlete Console](https://console.authlete.com/), not in code or env vars. The server dynamically reads these settings at runtime.

### Step 1: Enable FAPI profile

1. Log into [Authlete Console](https://console.authlete.com/)
2. Select your Service
3. Go to **Service Settings → Endpoints → Advanced → FAPI**
4. Under **Supported Service Profiles**, check **FAPI2_SECURITY**
5. Click **Save**

### Step 2: Create a scope with fapi2=sp attribute

1. Go to **Service Settings → Tokens and Claims → Advanced → Supported Scopes**
2. Click **Create**
3. Enter a scope name (e.g. `fapi_scope`)
4. Under **Attributes**, add an entry:
   - **Key**: `fapi2`
   - **Value**: `sp`
5. Click **Save**

### Step 3: Register a confidential client with PRIVATE_KEY_JWT

This is the most important step for FAPI compliance.

1. Go to **Clients → Create**
2. **Client Type**: `Confidential`
3. **Token Auth Method**: `PRIVATE_KEY_JWT`

   > FAPI 2.0 SP §5.3.2.1.6 requires `private_key_jwt` or `tls_client_auth`. Do NOT use `CLIENT_SECRET_POST` or `CLIENT_SECRET_BASIC` — Authlete will reject requests from FAPI scope requests if the client uses a weaker method.

4. **Grant Types**: `AUTHORIZATION_CODE`, `REFRESH_TOKEN`
5. **Redirect URIs**: `http://localhost:3001/callback`
6. **JWK Set**: Paste the public key generated by the SPA wizard (see Quick Start). The JWK Set format is:
   ```json
   {"keys":[{"kty":"EC","crv":"P-256","kid":"<kid>","x":"<x>","y":"<y>","alg":"ES256","use":"sig"}]}
   ```
7. **PAR**: Enable **Require Pushed Authorization Requests**
8. **PKCE**: Enable **Require PKCE** (S256)
9. Click **Save** and note the `clientId`

### Step 4: Configure DPoP

DPoP is configured at the **service level**, not per-client:

1. Go to **Service Settings → Tokens and Claims → Advanced → DPoP Token**
2. **Require Nonce**: `true` (set to `false` for simpler testing)
3. **Nonce Duration**: `3600` (1 hour)
4. Click **Save**

## Part 3: Step-by-Step FAPI 2.0 Flow

This section walks through a complete FAPI 2.0 authorization code flow with `private_key_jwt` client auth and DPoP sender-constrained tokens.

### Prerequisites

1. Authlete service configured with FAPI 2.0 Security Profile (see Part 2)
2. A scope with `fapi2=sp` attribute (e.g., `fapi_scope`)
3. A confidential client with `PRIVATE_KEY_JWT` auth method and registered JWK Set
4. DPoP enabled at the service level

### Overview of the flow

```
Client                              Authorization Server (Authlete)
  │                                       │
  │── 1. Generate client auth key pair ──→│  (private_key_jwt signing)
  │── 2. Generate DPoP key pair ─────────→│  (sender-constrained binding)
  │                                       │
  │── 3. PAR ────────────────────────────→│  Parameters pushed with
  │    (client_assertion + DPoP proof)     │  private_key_jwt client auth
  │←── 4. request_uri ────────────────────│
  │                                       │
  │── 5. Authorize redirect ─────────────→│  Login + consent
  │←── 6. Authorization code ────────────│
  │                                       │
  │── 7. Token request ──────────────────→│  client_assertion + DPoP proof
  │    (client_assertion + DPoP proof)     │
  │←── 8. Access token (DPoP-bound) ─────│
  │                                       │
  │── 9. Userinfo + DPoP proof ─────────→│  Proof includes ath claim
  │    (ath + nonce)                      │
  │←──10. Userinfo response ─────────────│
```

### Step 1: Generate client auth key pair

The client generates an ES256 key pair for `private_key_jwt` client assertions. The private key stays in the browser; the public key is registered as the client's JWK Set in Authlete Console.

```javascript
const signingKey = await generateSigningKeyPair();
// Register signingKey.publicKey as JWK Set in Authlete Console
```

### Step 2: Generate DPoP key pair

Separate from the signing key, the client generates a DPoP key pair for sender-constrained tokens:

```javascript
const dpopKey = await generateKeyPair();
```

### Step 3: Push Authorization Request (PAR)

The client creates a `private_key_jwt` assertion and a DPoP proof, then pushes the authorization request:

```javascript
// Create client assertion (private_key_jwt)
const clientAssertion = await createClientAssertion(
  signingPrivateKeyJwk,
  clientId,
  "http://localhost:3000/api/token"    // aud = token endpoint
);

// Create DPoP proof for the PAR endpoint
const parProof = await createProof(
  dpopPrivateKeyJwk,
  "POST",
  "http://localhost:3000/api/par",
  undefined,  // no ath yet
  undefined,  // first request, no nonce
);
```

HTTP request:

```http
POST /api/par HTTP/1.1
Content-Type: application/json
DPoP: <parProof>

{
  "parameters": "response_type=code&client_id=<clientId>&redirect_uri=<redirectUri>&scope=fapi_scope%20openid&code_challenge=<pkceChallenge>&code_challenge_method=S256&state=<state>&client_assertion_type=urn%3Aietf%3Aparams%3Aoauth%3Aclient-assertion-type%3Ajwt-bearer&client_assertion=<clientAssertion>"
}
```

Note: `client_assertion_type` and `client_assertion` are URL-encoded within the `parameters` string. The server forwards the entire `parameters` string to Authlete, which extracts and validates the assertion.

Response (success):

```http
HTTP/1.1 201 Created
DPoP-Nonce: <serverNonce>

{
  "requestUri": "urn:ietf:params:oauth:request_uri:<id>",
  "expires_in": 90
}
```

### Step 4: Authorize

Redirect the user:

```
GET /api/authorize?client_id=<clientId>&request_uri=urn:ietf:params:oauth:request_uri:<id>
```

The server shows login → consent → redirects back with authorization code.

### Step 5: Exchange code for token

After the callback receives the authorization code, the client exchanges it using `private_key_jwt` + DPoP:

```javascript
// Create fresh client assertion for token exchange
const tokenAssertion = await createClientAssertion(
  signingPrivateKeyJwk,
  clientId,
  "http://localhost:3000/api/token"
);

// Create DPoP proof for the token endpoint
const tokenProof = await createProof(
  dpopPrivateKeyJwk,
  "POST",
  "http://localhost:3000/api/token",
  undefined,  // no ath yet (no access token to bind to)
  latestNonce,
);
```

```http
POST /api/token HTTP/1.1
Content-Type: application/x-www-form-urlencoded
DPoP: <tokenProof>

grant_type=authorization_code&code=<authCode>&redirect_uri=<redirectUri>&code_verifier=<pkceVerifier>&client_id=<clientId>&client_assertion_type=urn%3Aietf%3Aparams%3Aoauth%3Aclient-assertion-type%3Ajwt-bearer&client_assertion=<tokenAssertion>
```

Response:

```http
HTTP/1.1 200 OK
DPoP-Nonce: <newNonce>
Cache-Control: no-store

{
  "access_token": "DPoP-bound-token",
  "token_type": "DPoP",
  "expires_in": 3600,
  "refresh_token": "refresh-token",
  "scope": "fapi_scope openid"
}
```

The `token_type: "DPoP"` confirms the token is sender-constrained.

### Step 6: Call userinfo with DPoP

Compute the `ath` (access token hash) and create a DPoP proof bound to the token:

```javascript
const ath = await computeAth(accessToken);

const userinfoProof = await createProof(
  dpopPrivateKeyJwk,
  "POST",
  "http://localhost:3000/api/userinfo",
  ath,
  latestNonce,
);
```

```http
POST /api/userinfo HTTP/1.1
Authorization: Bearer <accessToken>
DPoP: <userinfoProof>
```

Since the token was issued with `token_type: "DPoP"`, Authlete requires a valid DPoP proof for every API call using this token. The proof's `ath` claim binds it to the specific access token.

### DPoP within FAPI — how Authlete handles it

Authlete's DPoP processing is entirely server-side. When the server forwards a DPoP header to Authlete:

1. Authlete extracts the JWT from the `DPoP` header
2. Validates the signature against the public key in the `jwk` header
3. Checks `htm` matches the HTTP method
4. Checks `htu` matches the request URL
5. If `ath` is present, validates it matches the access token's SHA-256 hash
6. If a nonce is expected, validates the `nonce` claim
7. Returns `dpopNonce` in the response if a new nonce is issued

This means FAPI-compliant DPoP is **not about the client implementing complex crypto** — it is about including the `DPoP` header with a properly signed proof on every request. Authlete handles the hard parts.

### DPoP Nonce Flow

Nonces provide proof of liveness and prevent pre-generation of DPoP proofs:

| Scenario | Behavior |
|----------|----------|
| `dpopNonceRequired: true`, first request without nonce | 401 `use_dpop_nonce` + `DPoP-Nonce` header |
| `dpopNonceRequired: true`, valid nonce | Success, may return new `DPoP-Nonce` |
| `dpopNonceRequired: false` | Success without nonce, but server may still issue a nonce on success responses |
| Expired nonce | 401 `invalid_dpop_proof` + new `DPoP-Nonce` |

The client should store the most recent nonce from any endpoint and include it in subsequent DPoP proofs.

### Private_key_jwt assertion structure

The `private_key_jwt` client assertion is a signed JWT:

```json
// JWT Header
{
  "alg": "ES256",
  "kid": "<key-id>"
}

// JWT Payload
{
  "iss": "<client_id>",
  "sub": "<client_id>",
  "aud": "http://localhost:3000/api/token",
  "exp": 1700000300,
  "iat": 1700000000,
  "jti": "urn:uuid:..."
}
```

The assertion is valid for 5 minutes (300 seconds). A fresh assertion should be generated for each API call.

## Part 4: Client Demo Walkthrough

The React SPA includes a **FAPI 2.0 Security Profile** section that lets you run a complete FAPI 2.0 SP flow interactively.

### Opening the FAPI section

1. Start both servers: `npm --prefix server run dev` + `npm --prefix client run dev`
2. Open `http://localhost:3001`
3. Click **FAPI 2.0 Security Profile** in the sidebar

### FAPI tools

**1. Fetch Config** — Shows live FAPI mode and DPoP status:
- `mode`: `"sp"`, `"ms"`, or `"disabled"`
- `dpopEnabled`: DPoP nonce is enabled
- `requiredClientAuth`: always `"PRIVATE_KEY_JWT"` for FAPI configurations
- `senderConstrainedTokens`: `"DPoP"` or `"none"`

**2. Fetch Status** — Raw Authlete configuration including all flags

**3. DPoP Key Utilities** — Standalone DPoP proof generation for testing with any endpoint

### FAPI 2.0 SP Test Flow wizard

The wizard walks through the complete FAPI 2.0 SP flow:

**Setup:**
- Enter Client ID, Redirect URI, and scopes (include your `fapi2=sp` scope, e.g., `fapi_scope`)
- Click **Generate Client Auth Key** — creates an ES256 key pair for `private_key_jwt`
- Copy the displayed JWK Set and register it in Authlete Console under the client's JWK Set
- Click **Generate DPoP Key** — creates the DPoP key pair for sender-constrained tokens

**Step 1: Push PAR**
- The wizard generates a fresh `private_key_jwt` assertion and DPoP proof
- Sends the PAR request with both embedded in the `parameters` string
- Displays the PAR response with `requestUri` and `expiresIn`

**Step 2: Authorize**
- Redirects to the authorization page with the `request_uri` from PAR
- After login + consent, redirects to the callback page
- The callback page automatically generates a new `private_key_jwt` assertion + DPoP proof and exchanges the code for tokens

**Step 3: Call Userinfo**
- Uses the stored DPoP key and access token
- Computes `ath` from the access token
- Calls userinfo with a DPoP proof and displays the response

### Client-side architecture

The wizard uses two distinct key pairs:

| Key Pair | Purpose | Registered Where |
|----------|---------|-----------------|
| **Client Auth Key** (ES256) | Signs `private_key_jwt` assertions for client auth | Client's JWK Set in Authlete Console |
| **DPoP Key** (ES256) | Signs DPoP proofs for sender-constrained tokens | Embedded in each DPoP proof's `jwk` header |

Both keys are generated client-side using `crypto.subtle`. Private keys never leave the browser (stored in `sessionStorage` for the duration of the session).

## Part 5: Troubleshooting

### "FAPI mode shows disabled"

Authlete service does not have FAPI enabled. Go to Authlete Console → **Service Settings → Endpoints → Advanced → FAPI** → enable `FAPI2_SECURITY`.

### "dpopEnabled is false"

Authlete has `dpopNonceRequired` set to `false`. Go to **Service Settings → Tokens and Claims → Advanced → DPoP Token** → set Require Nonce to `true`. Alternatively, DPoP works without nonces — the flag only controls nonce enforcement, not DPoP itself.

### "FAPI mode shows sp but Authlete doesn't enforce FAPI rules"

Missing scope with `fapi2=sp` attribute. Authlete only enforces FAPI rules when a requested scope carries the attribute. Create a scope with `fapi2=sp` and include it in the request.

### "Invalid DPoP proof" errors

Common causes:
- `htm` does not match the HTTP method
- `htu` does not match the actual URL
- `ath` is wrong or missing
- `nonce` is wrong or missing
- DPoP key does not match the key used in the token request
- **ES256 signature is DER-encoded instead of raw R||S** — use raw IEEE P1363 format (64 bytes for P-256)

### "The DPoP header did not include a public key in JWK format"

The DPoP proof JWT header must include the full `jwk` member with the public key. The `kid` alone is insufficient per RFC 9449.

### "The client authentication method is 'client_secret_post' but..."

Your client is configured for `CLIENT_SECRET_POST` but you are using a different auth method (or vice versa). For FAPI 2.0 SP, the client must use `PRIVATE_KEY_JWT`. Update the client's Token Auth Method in Authlete Console.

### "Invalid_client" on token endpoint with private_key_jwt

Check that:
- The client's JWK Set contains the public key matching the assertion's `kid`
- The `aud` in the assertion matches the token endpoint URL configured in Authlete Console
- The assertion is not expired (> 5 minutes from `iat`)
- The `iss` and `sub` match the `client_id`

### "Not a DPoP bearer token" error

The token was issued without DPoP binding, but you're sending a DPoP proof. Ensure the token endpoint received a valid DPoP proof during the initial token request.

### "HTTP 400: Missing required body field: parameters"

The PAR endpoint requires a JSON body with a `parameters` field. The `parameters` string must contain URL-encoded OAuth parameters including `client_assertion_type` and `client_assertion`.

## Part 6: Failure Demonstrations

These demos prove that DPoP sender-constrained tokens actually prevent token theft. Run these with a DPoP-bound access token from the wizard.

### Demo 1: Stolen token without a DPoP proof

```bash
curl -v -X POST http://localhost:3000/api/userinfo \
  -H "Authorization: Bearer <YOUR_DPOP_TOKEN>"
```

Expected: `401` — Authlete detects the token was issued with `token_type: DPoP` and requires a valid DPoP proof.

### Demo 2: Stolen token with a different DPoP key

Generate a different key pair and create a proof with it:

```bash
# The thief's proof uses a different key than the one bound to the token
curl -v -X POST http://localhost:3000/api/userinfo \
  -H "Authorization: Bearer <YOUR_DPOP_TOKEN>" \
  -H "DPoP: <THIEF_DPOP_PROOF>"
```

Expected: `401 invalid_dpop_proof` — Authlete validates that the public key in the DPoP proof's `jwk` header matches the key bound to the token.

### Demo 3: Bearer token used with DPoP header

```bash
curl -v -X POST http://localhost:3000/api/userinfo \
  -H "Authorization: Bearer <BEARER_TOKEN>" \
  -H "DPoP: <SOME_DPOP_PROOF>"
```

Expected: `401` — "Not a DPoP bearer token."

### What DPoP prevents

| Attack Scenario | Result |
|----------------|--------|
| Token stolen from browser storage | ❌ Fails — private key required for DPoP proof |
| Token + attacker's own key pair | ❌ Fails — `jwk` in proof must match token binding |
| Bearer token used with DPoP | ❌ Fails — token_type must be DPoP |

### Security principles

1. **priv_key_jwt for client auth**: The client proves its identity by signing a JWT with its registered private key. Authlete validates the signature against the client's JWK Set.

2. **DPoP for token binding**: The client binds each API call to its access token using a DPoP proof. A stolen token cannot be used without the client's private DPoP key.

3. **Headers vs body**: DPoP fields (`dpop`, `htm`, `htu`) are always extracted from HTTP headers, never from the request body. This prevents a compromised client from overriding the binding.

4. **Separate keys for separate purposes**: The client auth key and DPoP key are independent. Compromising one does not compromise the other.
